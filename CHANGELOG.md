# История изменений

## 1.7.0 — 10.07.2026

### Финансовая корректность

- Первый платёж только процентами теперь считается дополнительным stub-периодом и не уменьшает число амортизирующих платежей; договорная дата закрытия сдвигается на один период без неожиданного крупного финального остатка.
- Централизованные пределы денежных сумм, процентов и итоговых агрегатов отклоняют `Number.MAX_VALUE`, переполнение, `NaN` и `Infinity`; каждая строка графика и итог сценария проверяются до передачи в UI и экспорт.

### Безопасность и целостность данных

- JSON import/export и share-link используют единый лимит 8 МиБ для UTF-8 данных и потоковую защиту распаковки; большое валидное состояние снова проходит round-trip и не экспортируется в заведомо неимпортируемый файл.
- Persisted revision и `updatedAt` обнаруживают более новую запись другой вкладки, блокируют молчаливую перезапись и предлагают явно загрузить внешнюю версию либо перезаписать её без автоматического merge финансовых данных.

### Производительность

- Проверка больших наборов операций заранее строит множество регулярных дат и больше не выполняет линейный проход по календарю для каждой операции; явный режим дополнительного платежа не запускает поиск регулярной даты.
- Расчёт четырёх сравнительных сценариев повторно использует уже проверенный набор данных и не запускает полную валидацию перед построением каждого графика.
- Полная проверка импортируемого кредита, включая построение графика, выполняется в отдельном Worker; основной поток проверяет только размер и структуру JSON-контейнера, а новая задача отменяет устаревшую.
- Объединённый список досрочных операций определяет ручные записи через `Set` ID вместо линейного поиска по всему массиву для каждой строки.
- Большой график отображается страницами по 100 строк без кнопки, монтирующей весь schedule; переход по дате открывает нужную страницу, а полный DOM отчёта создаётся только по запросу печати.

### Экспорт, импорт и восстановление

- JSON и share-link формируются из готового Worker-snapshot той же ревизии параметров; синхронный повторный расчёт удалён, а несовпадающая ревизия блокирует экспорт.
- JSON-импорт принимает только явно поддерживаемую числовую версию `1`; неизвестные указанные версии отклоняются с понятной ошибкой, а legacy-файлы без `version` по документированному правилу считаются версией `1`.
- Явно заданные повреждённые тип платежа, частота, округление, процентные правила, boolean-поля, режим ставки, amount mode и неизвестная валюта больше не заменяются молча; допустимые legacy-преобразования `RUR`, `total` и отсутствующего `amountMode` сопровождаются предупреждениями.
- Скрытие уведомления о карантине больше не удаляет raw-данные: скачивание backup, скрытие сообщения и необратимое удаление с подтверждением стали отдельными действиями.
- CSV, HTML/Excel и печать/PDF блокируются при отсутствующем, устаревшем или ошибочном результате расчёта; кнопки больше не создают пустой либо несогласованный отчёт.
- При ошибке расчёта доступен отдельно помеченный `Raw recovery JSON` с исходными параметрами и диагностикой; он не выдаётся за проверенный calculated export.

### Интерфейс и доступность

- Печатный отчёт явно помечает выключенные разовые платежи и регулярные правила как «Выключено», а не показывает их как применённые к расчёту.
- Карточка текущего долга названа «Общая текущая задолженность» и поясняет состав, тогда как отдельная карточка «Тело кредита» продолжает показывать только основной долг.
- Для будущего кредита карточка «Тело кредита» показывает запланированный principal после выдачи, а не нулевой текущий долг до даты выдачи.
- Подсказки даты и новой ставки учитывают режим `exactDate` и больше не утверждают, что ставка всегда начинает действовать только со следующего платёжного периода.
- Законный `finalBalloon`, возникающий из явно выбранных условий, показывается отдельным предупреждением с датой и суммой в Overview и печатном отчёте.
- Скрытая мобильная навигация получает `inert` и `aria-hidden`, поэтому её кнопки не остаются в tab order до открытия меню.
