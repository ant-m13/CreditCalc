# Публикация и релизы

Проект публикуется как статический сайт. Готовая сборка находится в папке `dist`.

## GitHub Pages

Workflow:

```text
.github/workflows/deploy-pages.yml
```

Запускается:

- при push в `main`;
- вручную через GitHub Actions.

Что делает:

1. включает pnpm через Corepack;
2. устанавливает зависимости;
3. запускает тесты;
4. собирает проект;
5. публикует `dist` через GitHub Pages.

Для GitHub Pages используется переменная:

```text
VITE_BASE_PATH=/${{ github.event.repository.name }}/
```

Она нужна, чтобы приложение корректно работало в подпапке репозитория.

## Автоматический тег версии

Workflow:

```text
.github/workflows/auto-release.yml
```

Запускается при изменении `package.json` в ветке `main`.

Что делает:

1. читает версию из `package.json`;
2. проверяет, что в `CHANGELOG.md` есть раздел для этой версии;
3. устанавливает зависимости;
4. запускает тесты;
5. собирает проект;
6. создаёт и отправляет тег `vX.Y.Z`, если такого тега ещё нет.

Этот workflow не создаёт GitHub Release. Он только создаёт тег.

## GitHub Release с архивом сайта

Workflow:

```text
.github/workflows/release-dist.yml
```

Запускается:

- автоматически при push тега `v*`;
- вручную, если указать существующий тег.

Что делает:

1. checkout указанного тега;
2. устанавливает зависимости;
3. запускает тесты;
4. собирает проект с `VITE_BASE_PATH=./`;
5. архивирует содержимое `dist`;
6. прикрепляет `credit-calculator-dist.zip` к GitHub Release.

`VITE_BASE_PATH=./` делает архив пригодным для размещения на любом статическом хостинге, а не только на GitHub Pages.

## Рекомендуемый порядок выпуска версии

1. Внести изменения в код и документацию.
2. Обновить версию в `package.json`.
3. Добавить раздел в `CHANGELOG.md`.
4. Прогнать проверки:

```bash
pnpm typecheck
pnpm test
pnpm build
pnpm audit --prod
```

5. Создать коммит и отправить `main`.

После push workflow создаст тег, а затем отдельный workflow создаст GitHub Release с архивом готового сайта.

## Ручной релиз

Если тег уже существует, можно вручную запустить `release-dist.yml` и указать тег, например:

```text
v1.5.2
```

Workflow соберёт проект именно из этого тега и обновит/создаст релиз с архивом `credit-calculator-dist.zip`.
