# Публикация и релизы

Проект публикуется как статический сайт. Готовая сборка находится в папке `dist`.

## GitHub Pages

Workflow:

```text
.github/workflows/deploy-pages.yml
```

Запускается:

- при push в `main`;
- вручную через GitHub Actions.

Что делает:

1. включает pnpm через Corepack;
2. устанавливает зависимости;
3. запускает ESLint;
4. запускает тесты;
5. собирает проект;
6. публикует `dist` через GitHub Pages.

Для GitHub Pages используется переменная:

```text
VITE_BASE_PATH=/${{ github.event.repository.name }}/
```

Она нужна, чтобы приложение корректно работало в подпапке репозитория.

## Автоматический релиз версии

Workflow:

```text
.github/workflows/auto-release.yml
```

Запускается при изменении `package.json` в ветке `main` или вручную.

Что делает:

1. читает версию из `package.json`;
2. проверяет, что в `CHANGELOG.md` есть раздел для этой версии;
3. устанавливает зависимости;
4. запускает ESLint;
5. запускает тесты;
6. собирает проект;
7. создаёт и отправляет тег `vX.Y.Z`, если такого тега ещё нет;
8. архивирует содержимое `dist`;
9. создаёт или обновляет GitHub Release с архивом `credit-calculator-dist.zip`.

Workflow создаёт Release сам, потому что tag push, выполненный стандартным `GITHUB_TOKEN`, не запускает следующий workflow автоматически.

## GitHub Release с архивом сайта

Workflow:

```text
.github/workflows/release-dist.yml
```

Запускается:

- автоматически при push тега `v*`;
- вручную, если указать существующий тег.

Что делает:

1. проверяет формат тега `vX.Y.Z`;
2. проверяет, что указанный тег существует в репозитории;
3. checkout указанного тега;
4. устанавливает зависимости;
5. запускает ESLint;
6. запускает тесты;
7. собирает проект с `VITE_BASE_PATH=./`;
8. архивирует содержимое `dist`;
9. прикрепляет `credit-calculator-dist.zip` к GitHub Release.

`VITE_BASE_PATH=./` делает архив пригодным для размещения на любом статическом хостинге, а не только на GitHub Pages.

## Рекомендуемый порядок выпуска версии

1. Внести изменения в код и документацию.
2. Обновить версию в `package.json`.
3. Добавить раздел в `CHANGELOG.md`.
4. Прогнать проверки:

```bash
pnpm lint
pnpm typecheck
pnpm test
pnpm build
pnpm audit --prod
```

5. Создать release-ветку, коммит и PR в `main`.

После merge release-ветки в `main` workflow создаст тег и GitHub Release с архивом готового сайта.

## Ручной релиз

Если тег уже существует, можно вручную запустить `release-dist.yml` и указать тег, например:

```text
v1.6.6
```

Workflow соберёт проект именно из этого тега и обновит/создаст релиз с архивом `credit-calculator-dist.zip`.
