# Архитектура проекта

Кредитный калькулятор — статическое React-приложение. Оно не использует backend, базу данных и внешние API для расчётов. Все данные остаются в браузере пользователя.

## Основной поток данных

```text
Параметры кредита / платежи / льготные периоды
        ↓
Zustand store с persist в localStorage
        ↓
expandRepaymentRules
        ↓
compareScenarios
        ↓
generateBaseSchedule
        ↓
интерфейс, график, печать, экспорт
```

## Ключевые части проекта

- `src/App.tsx` — общая компоновка приложения, переключение разделов, импорт/экспорт, загрузка ссылок, печать.
- `src/store.ts` — состояние приложения, список кредитов, активный кредит, настройки интерфейса и миграции localStorage.
- `src/loanEngine` — расчётное ядро: проценты, график, досрочные погашения, льготные периоды, сравнение сценариев и валидация.
- `src/repaymentRules.ts` — преобразование регулярных досрочных платежей в обычный массив `EarlyRepayment`.
- `src/importExport.ts` — проверка и нормализация JSON перед загрузкой.
- `src/shareCalculation.ts` — сериализация расчёта в сжатую ссылку и восстановление из неё.
- `src/components` — разделы и модальные окна интерфейса.
- `src/components/ErrorBoundary.tsx` — последний защитный слой от неожиданных runtime-ошибок интерфейса.
- `src/styles` — разделённые CSS-файлы по областям интерфейса.
- `CHANGELOG.md` — источник страницы “Что изменилось” внутри приложения.
- `vite.config.ts` — версия приложения, дата сборки и базовый путь для GitHub Pages.

## Хранение данных

Данные сохраняются через Zustand persist в `localStorage`.

Текущий ключ хранилища:

```text
ipoteka-calculator-v1
```

Название ключа сохранено для совместимости со старыми установками, несмотря на новое название приложения.

Текущая версия persisted-схемы — `8`. При загрузке старого состояния приложение нормализует параметры кредита, историю изменения ставки, разовые платежи, регулярные платежи и льготные периоды. Записи с невозможными датами или повреждёнными обязательными полями отбрасываются, чтобы старый `localStorage` не мог постоянно ломать интерфейс. Массивы старых записей дополнительно обрезаются до безопасных лимитов, чтобы повреждённое хранилище не подвешивало интерфейс.

Persist использует безопасный адаптер localStorage. Если браузер запрещает запись или хранилище переполнено, ошибка не ломает интерфейс: приложение показывает предупреждение и предлагает сохранить расчёт через JSON-экспорт.

Каждый кредит хранит собственные:

- параметры кредита и историю изменения ставки;
- разовые досрочные платежи;
- регулярные досрочные платежи;
- льготные периоды;
- выбранный сценарий;
- настройки отображения;
- тему и акцентный цвет.

## Сценарии расчёта

`compareScenarios` строит несколько вариантов:

- без досрочных платежей;
- сокращение срока;
- снижение платежа;
- по операциям.

Сценарий “по операциям” использует стратегию, указанную в каждой досрочной операции. В интерфейсе конкретной операции пользователь выбирает только прикладные стратегии: сократить срок, снизить платёж или закрыть кредит полностью.

## Импорт, экспорт и ссылки

JSON и ссылки не сохраняют готовый график платежей. Они сохраняют исходные данные, после чего график пересчитывается локально.

Это важно по двум причинам:

1. файл и ссылка остаются меньше;
2. после обновления расчётного ядра старые данные получают актуальный пересчёт.

Перед импортом JSON выполняется проверка структуры, дат, чисел, валюты, типов платежей, истории ставки, льготных периодов и регулярных платежей.

## Производительность

Тяжёлые разделы интерфейса загружаются лениво:

- обзор;
- настройки;
- досрочные платежи;
- график платежей;
- импорт/экспорт.

Это уменьшает основной JS-бандл и ускоряет первичную загрузку на статическом хостинге.

## Качество кода

Проект проверяется через TypeScript, ESLint, Vitest и production-сборку. ESLint включает правила TypeScript и React Hooks; тесты покрывают расчётное ядро, импорт/экспорт, шаринг ссылкой, миграции store и базовый UI-сценарий формы досрочного платежа.

## Безопасность

Приложение не отправляет финансовые данные на сервер. Основные риски связаны не с backend, а с действиями пользователя:

- JSON-файл может содержать конфиденциальные параметры кредита;
- ссылка на расчёт содержит все исходные данные;
- любой получатель ссылки сможет восстановить расчёт.

Поэтому в интерфейсе и README отдельно указано предупреждение про ссылки.

## Защита от повреждённых данных

Приложение проверяет данные на нескольких уровнях:

- JSON-импорт отклоняет повреждённые даты, неизвестные enum-значения и слишком большие массивы;
- расчётное ядро повторно проверяет входные данные перед построением графика;
- старые данные из `localStorage` нормализуются при миграции;
- при недоступном `localStorage` пользователь видит предупреждение и может сохранить данные вручную через JSON;
- Error Boundary показывает понятное сообщение вместо пустого экрана, если в интерфейсе всё же возникла непредвиденная ошибка.
